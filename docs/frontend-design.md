# ΩmegaWiki Standalone Local Frontend — Design

> Status: design draft (no code yet). Source-of-truth blueprint for a future `app/` (frontend) + `tools/serve.py` (backend) implementation.
> Reader: anyone who has not seen this conversation should be able to start building from this doc alone.

---

## 1. Context & Goals

ΩmegaWiki currently has three uncoordinated viewing surfaces:

- **Obsidian** — renders markdown well, but its graph view is rigid (just `colorGroups` + filters), can't show research-lifecycle state (claims by status, experiments running, open questions), and can't do entity-aware UI like "list all `weakly_supported` claims with `confidence < 0.4`".
- **`wiki/graph-view.html`** — a single Cytoscape page generated by `tools/visualize.py`. Graph-only. No reader, no dashboard, no editing, no inter-view navigation.
- **CLI** — `tools/research_wiki.py query`, `stats`, `maturity`, `neighbors`, `compile-context` expose rich queries, but only as terminal output.

**Goal**: a single local web app that unifies all three views (Reader, Graph, Dashboard), runs over a local Python API server, and stays consistent with the project's "Python + zero-Node" stack — vanilla JS plus CDN libs, no `npm` toolchain.

### Non-goals

- **Full `/skill` orchestration is out of scope.** Skills like `/ingest`, `/ask`, `/edit`, `/check`, `/exp-*` are Claude-Code-driven and require an LLM session. The frontend can directly call `tools/research_wiki.py` subcommands (CRUD on graph + frontmatter) and `tools/visualize.py`, but cannot execute multi-step skills end-to-end. UI buttons that need a skill must surface a "run `/skill X` in Claude Code" instruction — never silently fake the result.
- **Multi-user auth, public hosting, multi-tab write reconciliation** are out of scope for v1. Single-user local tool first.

## 2. Architecture overview

```
┌─────────────────────────────────────────────────┐
│  Browser (vanilla JS + CDN)                     │
│   ├─ Reader view   (markdown + backlinks)       │
│   ├─ Graph view    (Cytoscape, BFS, filters)    │
│   └─ Dashboard     (stats, lifecycle, log)      │
└────────────────────┬────────────────────────────┘
                     │ HTTP/JSON
                     ▼
┌─────────────────────────────────────────────────┐
│  tools/serve.py  (stdlib http.server + json)    │
│   ├─ GET  read endpoints (entities, graph,      │
│   │       stats, maturity, neighbors, log,      │
│   │       context_brief, open_questions)        │
│   ├─ POST write endpoints (proxied through      │
│   │       research_wiki.py CLI for validation)  │
│   └─ POST regenerate (proxied to visualize.py)  │
└────────────────────┬────────────────────────────┘
                     │ direct file I/O + subprocess
                     ▼
              wiki/, raw/, .checkpoints/
```

State lives only on disk. Backend is stateless. Browser fetches over HTTP. No WebSocket, no SSR, no DB.

## 3. Tech stack

| Layer | Choice | Rationale | CDN URL / module |
|---|---|---|---|
| Backend HTTP | `http.server` (stdlib) | Zero pip dependency. Sufficient for single-user local. | — |
| Backend write validation | subprocess `python tools/research_wiki.py <cmd>` | Reuses CLI's edge-type / confidence / status schema validation; avoids re-implementing. | — |
| Frontend graph | Cytoscape.js 3.28.1 | Already used in `tools/visualize.py:628`. | `unpkg.com/cytoscape@3.28.1/dist/cytoscape.min.js` |
| Markdown render | marked.js (latest) | Single-file, widely-used, supports custom renderer for `[[wikilinks]]`. | `cdn.jsdelivr.net/npm/marked/marked.min.js` |
| Frontmatter parse | js-yaml 4.1.0 | Required for body-vs-frontmatter split when serving raw `.md`. | `cdn.jsdelivr.net/npm/js-yaml@4.1.0/dist/js-yaml.min.js` |
| Routing | Hash router (~30 LOC, in-house) | Avoids any framework. | — |
| State | Plain ES modules + module-scoped objects | Avoids React/Vue/Svelte. | — |
| Search (optional, Phase 3) | MiniSearch 6.3.0 | Client-side full-text. | `cdn.jsdelivr.net/npm/minisearch@6.3.0/dist/umd/index.min.js` |

If `http.server` later proves limiting (streaming, auth, large file uploads), the documented migration target is **Flask** (single pip install, still close to stdlib feel). Defer that to an explicit "stack v2" decision.

## 4. API contract

Reference: subcommands surveyed in `tools/research_wiki.py` (lines 2750–2840). Every read endpoint is a thin shim over an existing function; most can be called as Python library functions to avoid subprocess overhead.

### Read (no auth, idempotent)

| Method | Path | Backed by | Returns |
|---|---|---|---|
| GET | `/api/stats` | `research_wiki.py stats --json` | wiki entity counts |
| GET | `/api/maturity` | `research_wiki.py maturity --json` | coverage score |
| GET | `/api/entities/{type}` | dir scan + frontmatter parse | array of `{slug, title, frontmatter}` |
| GET | `/api/entities/{type}/{slug}` | read file | `{frontmatter, body}` |
| GET | `/api/entities/{type}/{slug}/raw` | read file | raw markdown text |
| GET | `/api/graph` | `edges.jsonl` + `citations.jsonl` | `{nodes, edges, citations}` |
| GET | `/api/graph/neighbors/{node_id}?depth=N&edge_type=X` | `research_wiki.py neighbors` | sub-DAG |
| GET | `/api/find/{type}?field=...&value=...` | `research_wiki.py find` | array |
| GET | `/api/find-similar/{kind}?title=...` | `find-similar-concept` / `find-similar-claim` | candidates |
| GET | `/api/query/{subquery}?...` | `research_wiki.py query` | weak-claims / orphans / etc. |
| GET | `/api/index` | read `wiki/index.md` | text |
| GET | `/api/log?tail=N` | tail `wiki/log.md` | array of entries |
| GET | `/api/context-brief` | `graph/context_brief.md` | text |
| GET | `/api/open-questions` | `graph/open_questions.md` | text |

### Write (single-user local; no auth gate beyond loopback bind)

| Method | Path | Backed by | Notes |
|---|---|---|---|
| PATCH | `/api/entities/{type}/{slug}` | `research_wiki.py set-meta` | partial frontmatter update |
| POST | `/api/edges` | `research_wiki.py add-edge` | enforces edge-type schema |
| POST | `/api/edges/batch` | `research_wiki.py batch-edges` | bulk add via stdin pipe |
| POST | `/api/citations` | `research_wiki.py add-citation` | bibliographic |
| POST | `/api/log` | `research_wiki.py log` | append entry |
| POST | `/api/regenerate/index` | `research_wiki.py rebuild-index` | derived state |
| POST | `/api/regenerate/context-brief` | `research_wiki.py rebuild-context-brief` | derived state |
| POST | `/api/regenerate/open-questions` | `research_wiki.py rebuild-open-questions` | derived state |
| POST | `/api/regenerate/visualize` | `visualize.py generate-{obsidian-config,canvas,html}` | optional |

### Skill-trigger boundary endpoints (return guidance, do not execute)

| Method | Path | Returns |
|---|---|---|
| POST | `/api/intent/ingest` | `{message, command}` — copy-pasteable `/ingest <path>` snippet plus a one-line "this UI cannot orchestrate full ingest" note |
| POST | `/api/intent/ask` | similar |
| POST | `/api/intent/edit` | similar |

This makes the boundary explicit in the API surface itself — UI code calls these endpoints and renders the guidance, never silently fabricates skill output.

### Hard rules

- Bind to `127.0.0.1` only (no LAN exposure by default).
- All write endpoints validate via subprocess to inherit CLI schema enforcement (edge types, confidence ranges, claim status transitions).
- Read endpoints prefer direct Python library calls (`import research_wiki`) for latency; fall back to subprocess only if a function isn't importable.
- All responses are JSON except `/raw`, `/index`, `/context-brief`, `/open-questions` (text/markdown).

## 5. Backend layout (`tools/serve.py`)

Single file, stdlib only:

```
tools/serve.py
  class WikiHandler(http.server.BaseHTTPRequestHandler):
      do_GET    -> route to read handlers
      do_POST   -> route to write handlers
      do_PATCH  -> set-meta proxy (custom verb dispatch)
  Routing: dict[regex, handler_fn]
  Static: serve app/ as static files at /
  Helpers:
    subprocess_json(cmd, *args) -> parsed JSON stdout
    read_entity(type, slug) -> {frontmatter, body}
    list_entities(type) -> [{slug, title, frontmatter}, ...]
```

Run: `python tools/serve.py [--port 8765] [--host 127.0.0.1]`. Print URL on startup.

Static-file serving (the `app/index.html`, modules under `/app/`, plus assets pulled from CDN) means **one server, one port** — no CORS, no proxy fuss.

## 6. Frontend layout (`app/`)

```
app/
  index.html             # SPA shell, loads CDN libs, mounts <wiki-app>
  app.css                # Theme (light + dark via prefers-color-scheme)
  modules/
    api.js               # fetch wrappers, returns Promises of JSON
    router.js            # hash router: #/reader/{type}/{slug}, #/graph, #/dashboard
    state.js             # tiny pub/sub, no framework
    wikilink.js          # parse [[slug]] / [[slug|alias]] in markdown
    reader.js            # left tree | center md | right backlinks
    graph.js             # extracted Cytoscape + force-layout from visualize.py
    dashboard.js         # widgets: stats, claim-status pie, experiments table, log timeline, open questions
    components/
      entity-card.js     # used in graph hover and reader sidebar
      backlinks-panel.js
      log-timeline.js
      claim-status-pie.js
```

`graph.js` is **lifted directly** from the embedded HTML in `tools/visualize.py:_build_html_template()` (lines 611–1155). The custom force-directed layout (lines 712–832) and the Cytoscape style array (lines 904–977) port over unchanged.

Once `graph.js` is its own module, `tools/visualize.py generate-html` should be revisited so it stops embedding a giant inline JS string and instead emits a thin shell that loads `app/modules/graph.js`. This is **Phase 6 cleanup**, explicitly gated (Section 11).

## 7. View specs

### 7.1 Reader (`#/reader/{type}/{slug}`)

- **Left**: collapsible tree of all 9 entity dirs, each expanded into slug list, sorted by `importance` desc then alphabetically. Filterable by tag.
- **Center**: rendered markdown with:
  - `[[slug]]` links rewritten to `#/reader/{type}/{slug}` (resolve `type` by globbing all 9 dirs; ambiguous → disambiguation popover).
  - Frontmatter shown as a structured "metadata" card above body (importance badge, status pill, tags, dates).
  - YAML lists like `evidence`, `key_papers`, `linked_experiments` rendered as styled chip lists with click-to-navigate.
- **Right**: backlinks panel (which other pages link to this one — computed by indexing all body content client-side once on load, cached in memory). Plus a "graph context" button that opens the Graph view focused on this node at depth 2.

Required endpoints: `GET /api/entities/{type}`, `GET /api/entities/{type}/{slug}`, `GET /api/find` (for cross-references).

### 7.2 Graph (`#/graph?focus=...&depth=N&types=...&edge-types=...`)

Re-uses 100% of current `graph-view.html` interactivity:

- Cytoscape canvas with custom Obsidian-style force layout
- BFS depth slider
- Per-entity-type filter checkboxes
- Per-edge-type filter checkboxes
- Search box (slug + title)
- Click → highlight neighbours
- Double-click → **navigate to Reader** (`#/reader/{type}/{slug}`) instead of `obsidian://` (replacing the current behavior at `tools/visualize.py:1007–1013`)

New (SPA-only):

- Pin/unpin focus node
- Zoom-state persistence in `localStorage`
- "Save as canvas" button (calls `POST /api/regenerate/visualize` with `--focus`)

### 7.3 Dashboard (`#/dashboard`)

Widgets, top-down, each fed by one read endpoint:

1. **Headline strip**: paper count, concept count, claim count, edge count (from `/api/stats`).
2. **Maturity gauge**: cold/warm/hot from `/api/maturity`.
3. **Claims by status pie**: `proposed / weakly_supported / supported / challenged / deprecated` (from `/api/find/claims` aggregated client-side).
4. **Confidence histogram** for claims (10 bins).
5. **Experiments table**: status, target_claim, started, estimated_hours (from `/api/find/experiments`).
6. **Ideas pipeline**: kanban-style by status `proposed → in_progress → tested → validated/failed` (from `/api/find/ideas`).
7. **Open questions list**: from `/api/open-questions` (rendered markdown).
8. **Log timeline**: last 50 entries from `/api/log`, skill-icon-coded rows.
9. **Quick actions strip**: buttons that call the **intent** endpoints (Section 4) and surface "run this in Claude Code" guidance.

## 8. Wikilink resolution algorithm

Wikilinks `[[slug]]` may match files under multiple entity dirs (e.g. a paper and a concept could share a slug, though the project currently doesn't allow this). Resolution order:

1. On app load, fetch `/api/index` (or build a map by listing all 9 `/api/entities/{type}`) → in-memory `slug -> [{type, title}, ...]`.
2. On render, replace `[[slug]]` with anchor whose `href` is `#/reader/{type}/{slug}`.
   - If 1 match → link directly.
   - If >1 → render disambiguation chip that expands on click.
   - If 0 → render as red broken-link span (matches Obsidian convention).
3. Cache the slug map; invalidate on any `PATCH` response.

## 9. Implementation phases

| Phase | Scope | Deliverables |
|---|---|---|
| **0. Plumbing** | server skeleton + static serving | `tools/serve.py` returning hello-world JSON; serving `app/index.html` |
| **1. Reader (read-only)** | List + read entities, render markdown, resolve wikilinks | All `GET /api/entities/*`; `app/modules/{api,router,wikilink,reader}.js` |
| **2. Graph (read-only)** | Lift Cytoscape code from `visualize.py` into `app/modules/graph.js`; navigate to Reader on dbl-click | `GET /api/graph`; full Graph view with filters/BFS/search |
| **3. Dashboard** | All read widgets | All remaining read endpoints; `app/modules/dashboard.js` + components |
| **4. Write capability** | PATCH frontmatter; POST edges/citations/log; regenerate derived state | All write endpoints; in-app edit UI (frontmatter form, edge-add modal) |
| **5. Skill boundary UX** | Intent endpoints + "run in Claude Code" guidance components | `/api/intent/*` endpoints; copy-to-clipboard buttons |
| **6. Cleanup** | Make `tools/visualize.py generate-html` emit a thin shell that loads `app/modules/graph.js` | Refactor `_build_html_template`; update `/visualize` SKILL doc |

**Phases 0–3 are the MVP.** Phase 4+ is opt-in. Phase 6 is gated (Section 11).

## 11. Fallback / rollback strategy

The whole design is purely additive. If the SPA does not work out, every existing surface (Obsidian, `wiki/graph-view.html`, `tools/research_wiki.py` CLI, generated canvases) keeps working unchanged.

**Containment guarantees**:

- Only **two** new things land in the tree: `tools/serve.py` and `app/` (new directory). No existing file under `tools/`, `wiki/`, `config/`, `.claude/skills/` is modified during Phases 0–5.
- The frontend is read-mostly. Write endpoints (Phase 4) all go through the existing `research_wiki.py` CLI, which means schema validation is identical to what `/edit` and `/ingest` produce — there is no new wiki-format dialect to migrate off later.
- Each phase commit must be independently `git revert`-able.

**Rollback paths, ordered cheapest → strongest**:

1. **Stop using it** — don't run `tools/serve.py`. Obsidian + `graph-view.html` + CLI still work. No artifact is left running.
2. **Delete the additions** — `rm tools/serve.py && rm -rf app/`. Wiki content untouched.
3. **`git revert`** — revert the merge commits if already on `main`.
4. **Branch isolation** — Phases 0–5 must live on a feature branch (suggested: `feat/standalone-frontend`). Merge to `main` only after Phase 3 (MVP) passes verification and the user accepts the look/feel. Phase 4 (writes) can land on a second feature branch with a separate review gate.

**Phase 6 is the one risky step** because it modifies existing code:

- Phase 6 refactors `tools/visualize.py:_build_html_template` to emit a thin shell that loads `app/modules/graph.js`, eliminating the embedded JS duplicate.
- Phase 6 is **explicitly gated**: do not start until Phases 0–5 have shipped on `main` and the user has been using the SPA for at least one session without complaints.
- If Phase 6 is never done, the old single-file `graph-view.html` remains as a permanent independent fallback, at the cost of one duplicated codebase. That cost is acceptable.

**Recovery if Phase 6 lands and SPA later regresses**: `git revert` the Phase 6 commit. `tools/visualize.py` is the only file touched in Phase 6, so the revert is one file, no data migration.

## 12. Open questions

- **Should `app/` ship in the repo or be gitignored?** Recommend: ship in repo. Unlike `wiki/.obsidian/` (regenerated by `/init`), `app/` is hand-authored source code.
- **Auth model** when later exposed on a LAN: defer to a v2 design doc.
- **Does `tools/visualize.py generate-html` stay alongside the SPA or get retired?** Recommend: keep. The single-file HTML is useful for "send a snapshot to a colleague". After Phase 6, the SPA reuses the same `graph.js` module so they don't drift.
